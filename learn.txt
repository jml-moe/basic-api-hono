[Day5]

1) Resource pakai kata benda (NOUN), bukan kata kerja di URL
   - Salah: /get-products/
   - Benar: /products  (+ method GET)

2) HTTP method & status code (pakai sesuai arti)
   - 2xx sukses
     - 200 — berhasil standar
     - 201 — resource baru dibuat (biasanya setelah POST)
   - 4xx — kesalahan dari client
     - 401 — unauthorized (belum/tidak sah login atau token), ex: akses /revenues tanpa hak
     - 403 — forbidden (sudah dikenali tapi tidak boleh)
     - 404 — not found, ex: GET /products/6 padahal hanya ada 5
   - 5xx — error server
     - 500 — generic; jangan expose detail sensitif ke user
   - Catatan: status utama lewat HTTP status line; hindari “status sukses/gagal” redundan di body sebagai sumber kebenaran.

3) Versioning API
   - Contoh breaking change: field amount → totalAmount di /payments
   - Solusi: /v1/... vs /v2/... supaya client lama tidak putus.

4) Filtering, sorting, pagination (query string)
   - Contoh: /v2/products?minP=100000&maxP=1000000
   - Memudahkan client tanpa menambah endpoint baru untuk tiap variasi.

5) Response format — konsisten antar endpoint
   - Opsi A — array langsung:
     GET /v1/products → []
     GET /v1/users    → []
   - Opsi B — dibungkus (mudah ditambah meta nanti):
     GET /v1/products → { 
              "data": [] 
            }
     GET /v1/users → { 
              "data": [] 
            }
   - Salah — campur format (frontend harus parsing beda-beda):
     /v1/products → { ... }  vs  /v1/users → []

6) Nested URL — pakai secukupnya (debatable)
   - Terlalu dalam bisa membingungkan:
     /v1/products/:id/reviews/:id/authors
   - Lebih datar sering lebih jelas:
     GET /v1/products/:id
     GET /v1/reviews?productId=<id>
     GET /v1/authors?reviewId=<id>

7) Aksi bisnis (action) — biasanya POST + path “verb”
   - Update umum resource:
     PATCH /v1/orders/:id
   - Aksi eksplisit (jelas ke frontend / dokumentasi):
     POST /v1/orders/:id/cancel
     POST /v1/orders/:id/proceed


[DAY 6] 

1) Pola placeholder route (biasakan sebelum logika penuh)
    .get("/", (c) => {
      return c.json([]);
    })
    .get("/:id", (c) => {
      return c.json({});
    })
    .post("/", (c) => {
      return c.json({ message: "" }, 201);
    })
    .patch("/:id", (c) => {
      return c.json({});
    })
    .delete("/:id", (c) => {
      return c.json({ message: "" });
    });

2) Request: headers vs body
   - GET — umumnya hanya headers + query; tidak ada body.
   - POST, PUT, PATCH, DELETE — bisa pakai headers dan body.
   - Headers — metadata (auth token, content-type, dll.)
   - Body — payload data yang diproses server.

3) Jenis database (gambaran besar)
   - SQL — ada schema, tabular, ex: PostgreSQL
   - NoSQL — fleksibel skema, ex: MongoDB

4) ORM
   - Semacam abstraksi yg memudahkan engineer untuk menulis bahasa query dengan bahasa yg udh familiar.
   - Abstraksi: query dengan API bahasa host (TypeScript), bukan raw SQL everywhere.
   - Contoh: SQL SELECT * FROM users ↔ ORM prisma.user.findMany()
